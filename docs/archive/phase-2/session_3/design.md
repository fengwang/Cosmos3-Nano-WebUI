# Session 3 Design - Joint Validation on RTX 5090

Date: 2026-07-09

## Context

`GPU-S1` rebuilt `deploy/vllm-omni.Dockerfile` from public inputs and proved
T2I generation, but against the pre-fix checkpoint with a manual
index-removal workaround. `GPU-S2` fixed both `wfen/*` Hugging Face repos
(LFS layout + stale index) and re-pinned every in-repo reference, verified by
fresh clone/download, but never ran the fix through the from-source image.
Neither claim has been tested together. This session is the joint-validation
gate: fresh `hf download` at the `GPU-S2` revisions, through the `GPU-S1`
image, no manual workaround, plus a best-effort T2V smoke. Stakeholders:
the release owner (gate sign-off on failure), and any future session/operator
who reads `docs/model_setup.md` / `README.md` per-mode markings and trusts
them because they point at an evidence row (`project_contract.md` §7).

Constraints carried over from `project_contract.md` and
`session_3_contract.yaml`: fresh-verification discipline (INV-4), no
Dockerfile/HF-repo edits, no API/WebUI-visible code changes, every GPU claim
records hardware/driver/commit/revision/request/artifact/pass-fail (INV-8),
and the human gate fires only on a T2I/T2V failure or a drift-D1 recurrence —
not before starting.

## Goals / Non-Goals

**Goals:** prove FP8 and NVFP4 T2I end to end (direct and full-stack) from a
genuinely fresh download; attempt a T2V smoke and record the outcome either
way; leave `docs/model_setup.md`, `docs/release_checklist.md`, `README.md`,
`docs/evidence_map.md`, `docs/risk_register.md`, and `docs/eval_seed_cases.md`
consistent with whatever the evidence actually shows.

**Non-Goals:** `t2v_audio`, `i2v`, `forward_dynamics`, `reasoning`, 720p
video (all PRD Non-Goals or explicit `session_3.md` out-of-scope items); any
change to `deploy/vllm-omni.Dockerfile` or either `wfen/*` HF repo; closing
R-10 (guardrails-on path — explicitly deferred, see `brainstorming.md`).

## Decisions

### D1 — Probe architecture: small focused probes + a pure aggregator

Already decided in brainstorming. Restated here with the concrete module
layout, since this is the part of the session that involves writing new
code and the functional-thinking skill applies directly.

**ACD classification of the new code** (`docs/session_3/probes/`):

| Unit | ACD Class | Reasoning |
|---|---|---|
| `EvidenceRecord`, `Verdict` | Data | inert facts describing one probe's outcome |
| `check_no_lfs_pointers(files)` | Calculation | same file-info tuple -> same verdict, always |
| `check_valid_image(bytes, expected_dims)` | Calculation | same bytes -> same verdict, always |
| `check_job_terminal(status_history)` | Calculation | same status sequence -> same verdict |
| `check_image_freshness(dockerfile_mtime, image_created_at)` | Calculation | explicit timestamps as params, not read internally -> pure |
| `fetch_checkpoint(repo_id, revision, local_dir)` | Action | shells `hf download`; network + filesystem |
| `list_checkpoint_files(local_dir)` | Action | filesystem walk, can change between calls |
| `bring_up_stack` / `tear_down_stack` | Action | `docker compose up`/`down`; process + host-state side effects |
| `post_direct_generation` / `post_fullstack_generation` | Action | HTTP calls to a live service |
| `poll_job_until_terminal` | Action | repeated HTTP GET, depends on wall-clock progress |
| `download_artifact` | Action | HTTP GET of generated bytes |
| `write_evidence_fragment` / `aggregate.py`'s file I/O | Action | filesystem write |
| `aggregate.py`'s fragment-merge logic itself | Calculation | same set of fragment dicts -> same merged dict, always |

**Public interface.** Each `run_*.py` exposes exactly one CLI entrypoint
(`python run_direct_t2i.py --checkpoint {fp8,nvfp4}`) that hides the compose
lifecycle, HTTP details, and evidence-writing — the common case (run this
task, get an evidence fragment) stays a single command. The pure checkers in
`lib.py` are the only part a caller needs to read to understand *what counts
as passing*; the action scripts are the only part that needs to know *how*
to gather the raw data.

**Data flow** (one task, `run_direct_t2i.py --checkpoint fp8` as the
example):

```
argv --checkpoint fp8
  -> resolve_request(checkpoint) -> RequestSpec        (Calculation)
  [Action: bring_up_stack(fp8 compose file)]
  [Action: post_direct_generation(url, RequestSpec) -> raw image bytes]
  -> check_valid_image(bytes, expected_dims) -> Verdict (Calculation)
  -> build_evidence_record(RequestSpec, Verdict, ...) -> EvidenceRecord (Calculation)
  [Action: write_evidence_fragment(path, EvidenceRecord)]
  [Action: tear_down_stack(fp8 compose file)]
```

**Error strategy** (pyramid tier per fallible operation; masking/retry stays
at the Action shell, never inside a Calculation):

| Operation | Failure mode | Tier | Residual contract |
|---|---|---|---|
| `fetch_checkpoint` | `hf download` exits non-zero | Propagate | Fragment written with `Verdict.FAIL`, `notes` = stderr tail; T1 fails -> human gate |
| `bring_up_stack` | compose build/up error, port conflict | Propagate | Fragment written with `Verdict.FAIL`, `notes` = compose output; T2I task fails -> human gate |
| `post_direct_generation` / `post_fullstack_generation` (T2I) | HTTP error, timeout | Propagate | `Verdict.FAIL` -> human gate |
| `post_direct_generation` (T2V smoke only) | HTTP error, timeout, VRAM OOM | Mask-as-data | `Verdict.SCOPED_OUT`, `notes` = reason; no human gate (SHOULD, not MUST) |
| `poll_job_until_terminal` | job never reaches a terminal state in budget | Propagate | `Verdict.FAIL`, `notes` = "no terminal state within `<n>`s" |
| `download_artifact` / `check_valid_image` | corrupt or invalid artifact | Propagate | `Verdict.FAIL`, `notes` = validity-check reason |
| `write_evidence_fragment` | disk write error | Propagate (uncaught) | Not a validation outcome — Failure Arbiter classifies as ENVIRONMENT, not BUG |

Every `run_*.py` has exactly one narrow `except Exception` boundary around
its action+check sequence: an unexpected exception becomes a
`Verdict.FAIL` (or, for the T2V script only, `Verdict.SCOPED_OUT`) evidence
fragment with the exception text in `notes`, never a bare crash and never a
silently swallowed `pass`. This directly satisfies "a T2V failure is
silently dropped instead of recorded" from the contract's adversarial cases.

**Parallelism assessment.** No parallelism opportunity identified — each
probe performs one sequential GPU/network operation, not a loop over a
collection. T3/T4 and T5/T6 additionally *must* run sequentially, not
concurrently, because the FP8 and NVFP4 compose overlays share a fixed
container name (`cosmos3-nano-webui-vllm-omni`) and only one stack can be up
at a time.

### D2 — Evidence schema matches INV-8 exactly

```python
@dataclass(frozen=True)
class EvidenceRecord:
    task_id: str                    # e.g. "T3-direct-t2i-fp8"
    hardware: str                   # "NVIDIA GeForce RTX 5090"
    driver_cuda: str                # from `nvidia-smi` header
    checkpoint_repo: str            # e.g. "wfen/Cosmos3-Nano-FP8-Blockwise"
    checkpoint_revision: str
    vllm_omni_commit: str           # must be 697035018b70cef76b974a909d23371a9984c3f2
    request_shape: dict             # mode, prompt/fixture, dims, seed, frames, fps, steps
    artifact_path: str | None
    artifact_metadata: dict | None  # dims, format, byte size
    verdict: Verdict                # PASS | FAIL | SCOPED_OUT — never a bare bool
    notes: str                      # required (non-empty) when verdict != PASS
```

`Verdict` is a 3-value enum, not a bool, because the T2V outcome is
genuinely tri-state (pass / fail / explicitly-scoped-out per
`session_3.md`'s SHOULD wording) and collapsing that to a bool would force a
caller to guess what `False` means (boolean blindness, functional-thinking
Check 5).

### D3 — Fetch target and image-freshness check

Fresh checkpoints land at `models/Cosmos3-Nano-{FP8,NVFP4}-Blockwise`
(repo-root-relative), matching the existing compose bind-mount path exactly —
no compose file changes needed (compose files are outside this session's
allowed edits anyway). Before T3, a pure `check_dockerfile_unmodified`
checks whether `deploy/vllm-omni.Dockerfile` has an uncommitted diff against
`HEAD` (gathered by a thin Action wrapper around `git diff --quiet`); if
dirty, stop rather than silently rebuild (this session cannot edit that
file, so a dirty diff means something unexpected happened outside this
session's control). If clean, reuse the cached `cosmos3-nano-vllm-omni:local`.

Originally designed as an mtime comparison (Dockerfile mtime vs. the
image's `Created` timestamp); rejected during T2 execution because
`git checkout` bumps a file's mtime on branch switch even when its content
is byte-identical — confirmed live when switching to `GPU-S3` made the
Dockerfile look newer than the image despite an identical sha256. A
content-identity signal (git diff, not mtime) is the correct check in a
git-managed repo.

### D4 — T2V checkpoint order and budget

NVFP4 attempted first (lower bit-width leaves more VRAM headroom for the
temporal dimension), lowest documented resolution (256, per
`docs/model_setup.md` §9's supported-dimensions set), minimal frame count.
If NVFP4 does not fit, FP8 is tried once; if neither fits, `Verdict.SCOPED_OUT`
is recorded with the OOM/error detail as `notes` — this satisfies the
deliverable ("recorded either way") without treating a VRAM-driven scope-out
as a session failure.

## Risks / Trade-offs

- **[Risk] A T2I failure blocks the whole session on the human gate** ->
  **Mitigation:** each task's evidence fragment is independent and already
  written before the failure is surfaced, so no work is lost; the owner
  decides retry vs. investigate vs. accept-as-limitation with full evidence
  in hand, not a bare error message.
- **[Risk] Fresh ~100 GB download + multiple container lifecycles could run
  long** -> **Mitigation:** downloads and long-running compose/generation
  calls run via backgroundable commands with explicit polling, not a single
  blocking foreground call; each task is checkpointed (evidence fragment
  written) before moving to the next, so a mid-session interruption loses at
  most one task's progress.
- **[Risk] Reusing a stale cached `cosmos3-nano-vllm-omni:local` image
  silently invalidates the "from-source, current-Dockerfile" claim** ->
  **Mitigation:** D3's explicit freshness check, not an assumption that the
  cached tag is current.
- **[Trade-off] Five small probe files instead of one script** is more
  files to read, but isolates failure blast radius per task and matches the
  Task Loop's one-task-at-a-time checkpoints — judged worth the extra
  file count for a high-risk session (accepted in brainstorming).

## Migration Plan

Not applicable in the deploy/rollback sense — this session adds
documentation, evidence, and session-local probe scripts; it does not change
any runtime code path. "Rollback" is simply not merging the `GPU-S3` branch

`docs/session_3/**` and the explicitly listed docs is touched until the
evidence exists to justify the change.
