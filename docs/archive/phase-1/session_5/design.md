# Session 5 Design - CPU-Only CI and Test Stabilization

Date: 2026-07-07
Session: MIG-S5
Risk: medium · Routing: worker_plus_reviewers · Gate: `GATE-MIG-S5-CI`

## Context

No `.github/` exists; CI is greenfield. The imported tree (S3) is CPU-safe by
design and `docs/model_setup.md` (S4) is the checkpoint contract. Constraints:
Python is pinned `>=3.12,<3.13` while the host interpreter is 3.14, so `uv` must
provision the toolchain; the heavy `oracle` extra (torch/diffusers) is opt-in and
must stay out of CI; `webui` is Next 15 / React 19 with pnpm 11 and a generated
`schema.d.ts`. The governing invariants are INV-5 (deterministic CPU CI, no CUDA /
weights / private network), INV-9 (no public API shape change), and INV-10 (every
added dependency justified and verified).

## Goals / Non-Goals

**Goals.** A CPU-only workflow that (1) runs Python lint + CPU tests, WebUI
lint/typecheck/unit tests, and both schema-sync layers; (2) is reproducible locally
from a documented command set; (3) needs no secrets, CUDA, weights, or self-hosted
runners; (4) cannot pass hollow (mass-skipped), and (5) is forward-safe against a
future GPU test breaking CPU collection.

**Non-Goals.** GPU CI, Docker image publishing, registry credentials, runtime
inference, `README.md`, and Docker/Compose render checks (no `deploy/` yet → S6).

## Decisions

- **D-1 · One `ci.yml`, two parallel jobs.** A single workflow with `python` and
  `webui` jobs gives one status context and parallel wall-clock. *Alternative:*
  split files (rejected — duplicated setup, two contexts). *Alternative:* one
  sequential job (rejected — serial, poor toolchain isolation).
- **D-2 · `uv`-managed Python 3.12, `--frozen`, `--group test-cpu`.** `setup-uv`
  provisions the pinned interpreter and installs from `uv.lock`. `--frozen`
  guarantees the lockfile is authoritative (R-10). We install core + `dev` +
  `test-cpu` and never `--extra oracle`, keeping torch out. *Alternative:* system
  Python + `pip` (rejected — host is 3.14, no lockfile determinism).
- **D-3 · CPU test extra to defeat hollow pass.** `numpy`/`pillow`/`imageio`/
  `imageio-ffmpeg`/`safetensors` are all torch-free and already pinned in the
  `oracle` extra; adding them as a `test-cpu` dependency group turns every
  dependency-gated `importorskip` skip into a real execution — the CPU suite goes
  from 467 passed / 3 skipped to 471 passed / 0 skipped (`EV-MIG-IMPORT-COMPLETE`). *Alternative:* accept the skips
  (rejected — weakens the gate); *Alternative:* add them to `dev` (rejected —
  keeps the host loop minimal and states intent).
- **D-4 · `next build` before `typecheck`.** `webui/tsconfig.json` includes
  `.next/types/**/*.ts`, generated by `next build`; building also validates the
  production bundle (three.js / urdf-loader). The order is `gen:api → build → lint
  → typecheck → test`. The dependency is confirmed empirically before finalizing;
  if `typecheck` passes standalone, the build still stays as a meaningful bundle
  check (recorded, not silently dropped). *Alternative:* lint/typecheck/test only
  (kept as fallback if build proves infeasible on the runner — would be a
  `SPEC_GAP` escalation).
- **D-5 · Schema sync = two existing torch-free layers.** No new tooling: the app
  side is `tests/test_openapi.py` (runs in the python job), the type side is
  `pnpm gen:api` + `git diff --exit-code`. This closes the "schema drift without
  failing CI" adversarial case at both boundaries.
- **D-6 · Scrub as a pure function under `tests/`.** `tools/**` is outside the
  blast radius, so the committed scan lives at `tests/test_private_ref_scan.py`,
  structured per functional-thinking: a pure `scan(paths, rules) -> findings`
  Calculation (regex over text), with file reads and the exit decision as thin
  Actions at the edges, plus a `__main__` CLI for local use. It runs inside the
  python `pytest` job. Patterns follow `docs/session_1/scrub_checklist.md`
  (high-confidence secrets: private-key headers, `hf_…`/`sk-…` tokens; committed
  weight/media file extensions; known-private tokens), with allowed `/path/to/…`
  placeholders and self-exclusions so the scanner and checklist do not trip it
  (`EV-MIG-SCRUB-COMMAND-SANITY`). *Alternative:* inline `rg` in YAML (rejected —
  not unit-testable, `rg` not preinstalled on runners); *Alternative:* gitleaks
  action (rejected by owner — third-party dependency).
- **D-7 · GPU isolation = marker + collection hook + `-m "not gpu"`.**
  `tests/conftest.py` implements `pytest_collection_modifyitems` to attach a skip
  to any `gpu`-marked item unless `COSMOS3_ENABLE_GPU_TESTS` is truthy. CI also
  passes `-m "not gpu"`. Two independent mechanisms mean neither an omitted CI flag
  nor a future GPU test can let GPU work run in CPU CI. The documented convention
  ("gpu modules guard heavy imports with `importorskip`") addresses the *collection
  import* failure mode the marker cannot (`-m` still imports modules to read
  markers). *Alternative:* CI flag only (rejected — single point of failure).
- **D-8 · Minimal, blast-radius-legal `ruff` fixes.** `F541` deletes a stray `f`;
  `E402` gets an inline `# noqa: E402` with a one-line reason (the file must inject
  `sys.modules` stubs before importing) rather than a global rule relaxation, so
  the ignore is local and self-documenting.
- **D-9 · Pinned, least-privilege workflow.** `permissions: contents: read`; no
  `secrets` referenced (R-14); actions pinned; `concurrency` cancels superseded
  runs; `NEXT_TELEMETRY_DISABLED=1`. Triggers `push` + `pull_request`.

## Risks / Trade-offs

- **`next build` needs network / is slow (fonts, telemetry, deps).** → Telemetry
  disabled; verify empirically on the build host; if flaky, classify `ENVIRONMENT`
  and record, do not mutate product code.
- **Scrub false positives block CI (`EV-MIG-SCRUB-COMMAND-SANITY`).** → Scope to
  the change-controlled surface, allow documented placeholders, exclude the
  scanner + checklist, and gate on "zero findings on the current clean tree" before
  relying on it. The scanner ships with tests proving both a clean pass and a
  synthetic-secret catch.
- **CPU CI green while GPU inference is broken (R-05).** → CI is explicitly
  CPU-only and makes no GPU claim; the manual GPU command and R-05 are handed to
  S8. Not solvable in-session by design.
- **Lockfile drift when adding `test-cpu` (R-10).** → `uv lock` regenerates
  `uv.lock`; CI uses `--frozen` so any future drift fails fast.
- **Node 26 (host) vs Node 22 (CI) skew.** → Pin CI to Node 22 LTS; local devs on
  newer Node are advised via the command list. Acceptable: the lockfile, not the
  Node minor, determines dependency resolution.

## Migration Plan

1. Land the `ruff` fixes and `test-cpu` group; confirm `ruff check api tests` clean
   and the three encoder tests execute.
2. Add `conftest.py` (GPU guard) and `test_private_ref_scan.py`; confirm clean-tree
   zero findings + synthetic catch + `-m "not gpu"` still passes.
3. Establish the WebUI baseline; confirm the `build → typecheck` dependency.
4. Author `ci.yml`; validate structure locally (mirror each step by hand;
   optionally `actionlint`).
5. Full deterministic checks → sharded review → adversarial verification → handoff.

Rollback: the workflow and test-harness additions are isolated; reverting the
commits removes CI without touching runtime source (no API/schema behavior change).

## Open Questions

- Does `pnpm typecheck` pass without a prior `next build` on the runner? Resolved
  empirically in step 3; either result keeps `build` as a bundle check.
- Should CI later assert a minimum collected-test count as an extra anti-hollow
  guard? Deferred — the `test-cpu` extra + `test_openapi` + scrub already make a
  hollow green implausible; revisit if S8 wants a hard floor.
